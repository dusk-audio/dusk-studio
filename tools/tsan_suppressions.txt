# ThreadSanitizer suppressions for the Dusk Studio Catch2 suite.
#
# Format: one rule per line. Patterns documented at
# https://github.com/google/sanitizers/wiki/ThreadSanitizerSuppressions
#
# These suppressions cover JUCE-internal patterns that TSan flags but
# that are NOT Dusk Studio bugs. The runtime continues but skips the
# matching warning. Project code is still fully instrumented.

# juce::WaitableEvent::signal() recursive-locks its internal mutex
# under heavy contention from juce::Thread::notify(). The pattern is
# inside JUCE 8.0.4 (juce_WaitableEvent.cpp:71 + juce_Thread.cpp:307)
# and not something Dusk Studio code controls. Upstream issue.
#
# Both `mutex:` (double-lock) and `deadlock:` (lock-order-inversion)
# suppression types needed — TSan reports the same JUCE pattern as
# either category depending on which thread hits it first.
mutex:juce::WaitableEvent::signal
mutex:juce::Thread::notify
deadlock:juce::WaitableEvent::signal
deadlock:juce::Thread::notify

# dusk::AutoResetEvent replaced juce::WaitableEvent in the DSP worker pool
# (de-JUCE threading migration). signal() now notifies while holding the mutex
# (the data-race fix), so the real synchronisation is clean; these stay as a
# backstop for the residual mutex/lock-order artifact TSan can still report for
# a mutex + condition_variable auto-reset primitive under heavy contention (same
# class as the juce::WaitableEvent entries above). Worker-pool correctness is
# covered by the join/quiesce protocol tests and the Parallel-Matches-Serial
# self-test.
mutex:dusk::AutoResetEvent::signal
deadlock:dusk::AutoResetEvent::signal

# JUCE's TimeSliceThread + ThreadedWriter pairing (used by Record
# Manager) ride the same notify path and trip the same warning. The
# RT-safety of Dusk Studio's own RecordManager code is verified by
# the bounded spin + null guards in src/engine/RecordManager.cpp
# (Phase 1).
mutex:juce::TimeSliceThread
deadlock:juce::TimeSliceThread

# juce::AbstractFifo's internal compare-exchange occasionally fires
# a TSan "data race" between the producer's relaxed store + the
# consumer's relaxed load. The lock-free FIFO contract holds — both
# sides intentionally use relaxed ordering for ring-buffer indices —
# and TSan's happens-before model doesn't recognise the algorithmic
# guarantee. Suppress + rely on the IPC self-test for correctness.
race:juce::AbstractFifo
